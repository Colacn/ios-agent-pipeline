# 测试纪律（Test）

> 借鉴 test-driven-development：Prove-It、证据优先、避免无效重复验证。  
> 与 `report-and-ledger-checklist.md`、`manual-intake.md` 配合。

## Prove-It（缺陷类）

修 bug 或回归时优先顺序：

```text
复现步骤/失败用例 → 确认失败 → 修复 → 同一用例转绿 → 全量回归
```

报告里须写：**复现证据**（命令/截图/日志片段）与**修复后同一证据**。

## 验证命令纪律

- 每次**可能改变结果的改动后**跑相应命令
- **代码未变**不要连续刷同一命令「求安心」
- 结论写清：跑了什么、版本/分支、通过还是未覆盖

## 测试金字塔（落盘口径）

- 单元/纯逻辑为主；集成测边界；E2E 只保关键路径
- 未测范围必须在 `test.report.md` **显式列出**，不得暗示「全测了」

## 状态优于交互

断言**结果状态**，少绑内部调用顺序；重构后测试仍应有效。

## 常见借口

| 借口 | 事实 |
|------|------|
| 「手动试过够了」 | 手动证据要写入报告，且不可复现等于未测 |
| 「以后补测试」 | develop 出口应已带最小回归集 |
| 「全绿就行」 | 须说明测了什么、没测什么 |

## 红旗

- 无改动说明却给「通过」
- bug 修复无复现用例或步骤
- 报告缺执行时间与命令
- 跳过 test 却无跳过理由与残余风险

## 出口核对

- [ ] `test.report.md` 或 `test.ledger.md` 已落盘
- [ ] 结论可复现；未覆盖范围已写
- [ ] 阻塞项已 AskQuestion 或标「等待用户答复」
