# 评审纪律（Review）

> 借鉴 code-review-and-quality：多轴评审、先看测试、评论分级、变更体量。  
> 与 `ruling-criteria.md`、`review-gate-checklist.md` 配合；本 skill 评的是 **analyze/plan 方案**，非代码 PR。

## 五轴覆盖（方案阶段）

| 轴 | 要问什么 |
|----|----------|
| **正确性** | 方案是否覆盖 analyze 需求与验收？边界与错误路径？ |
| **可读性** | 另一人能否无口述理解分层与改动范围？ |
| **架构** | 是否符合 overlay / 附录 A？有无不必要新范式？ |
| **安全** | 鉴权、输入、数据出境、密钥与合规 |
| **性能/可测** | 热点、批量、可观测性；测试策略是否可执行 |

## 评审顺序

1. **理解意图**：本方案解决什么？对应哪几条需求？
2. **先看验证故事**：plan 里的测试/验证命令是否具体？能否复现？
3. **再走五轴**：每条发现带证据锚点（inputs/outputs 路径或引用）
4. **分级输出**：与 `ruling-criteria.md` 的 Critical/High/Medium/Low 对齐

## 方案体量（类比代码 PR）

| 变更规模 | 建议 |
|----------|------|
| 聚焦单特性、改动面清晰 | 易评审 |
| 跨多子系统、无阶段切分 | 要求拆 plan 或标「分阶段评审」 |
| 无法在一轮内说清验收 | **不通过**，回 **plan** / **analyze** |

## 评论口径（写入 review.gate.md 时）

| 前缀 | 含义 |
|------|------|
| （默认） | 必改 |
| **Critical:** | 阻塞合入 |
| **建议:** | 可选优化 |
| **FYI** | 供后续参考，非必改 |

## 批准标准

在证据充分前提下：**明显增进整体方案质量即可通过**，不要求「完美方案」。与现有口径冲突时，以项目 overlay 与用户已决项为准。

## 常见借口

| 借口 | 事实 |
|------|------|
| 「能跑就行」 | 未评安全/可测/架构的「通过」会把债留给 develop |
| 「测试以后补」 | plan 阶段就要有可执行验证路径 |
| 「以后再澄清」 | 阻塞项须在闸门关闭或显式「暂不回答」 |

## 红旗

- 无证据的「通过」
- 安全/合规敏感却无对应章节
- plan 与 analyze 需求无法逐条对表
- 必改与建议混在一起、作者不知改什么

## 出口核对

- [ ] 结论：通过 / 有条件通过 / 不通过
- [ ] Critical 已闭环或触发不通过
- [ ] 验证故事可复现（命令/步骤写全）
