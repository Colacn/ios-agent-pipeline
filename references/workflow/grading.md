# 复杂度分级与默认路由

> 本文为 `AGENTS.md` 对应章节的结构化拆分镜像（第一阶段：复制下沉，不删主文档）。

## 默认工作流分级

- **L0 理解/定位**：只读分析、搜索与澄清，不改代码。
- **L1 小改动**：局部理解 -> 最小改动 -> 运行最小验证 -> 人工看 diff。
- **L2 多文件 / 跨模块**：先出方案或至少先明确落层、影响面、验收，再动代码。
- **L3 架构 / 高风险 / 发布相关**：按 `analyst -> planner -> reviewer -> developer -> tester` 全链路推进。

## 执行口径

> 各 IDE 入口见 [ide-adapters.md](../guide/ide-adapters.md)。

- `L0`：默认停在 `analyze/analyst`（不进入实现）
- `L1`：优先轻量快速通道（最小改动 + 最小验证）
- `L2`：默认先 `planner` 明确落层、影响面、验收再实现
- `L3`：默认走全链路流水线

## 何时算走流水线

满足任一即进入流水线：

- 用户显式委派子代理或 IDE 等价能力（如 `Task(subagent_type=…)`、`/planner` 等）使用流水线角色
- 用户使用 `/analyst`、`/planner` 等角色入口
- 用户明确要求按全链路 / 可评审交付推进
- 当前任务判断为 `L3`

## 硬例子（路由判定）

> 抽象分级对照表；与上文 L0–L3 口径一致，用于快速判定「走轻量还是全链路」。

| 场景 | 级别 | 路由 |
|------|------|------|
| 只读定位、解释代码 | L0 | `analyze`，不改代码 |
| 改 1 个文件文案/颜色 | L1 | 轻量通道（最小改动 + 最小验证），不走全链路 |
| 跨 IMKit + IMLib，动持久化 | L3 | 全链路：`analyze → plan → review → develop → test` |
| 用户输入 `/planner` 或 `Task(planner)` | 强制流水线 | 从 plan 或更早阶段（缺 analyze 则先补） |
| 仅优化 `.cursor` 规则文档 | L1–L2 | 可直改 + 自检，不必 fake 六阶段产物 |
