# Test 执行细则

> 本文件为 **test** Skill 的完整执行规范。与主会话或子代理调用方式无关，执行时以本文为准。

## 定位

把“改了啥”转为可验证、可追溯、可支撑放行决策的测试结论。

## 流水线中的本步（验证与放行）

- 位于 `develop` 之后，负责验证与放行建议。
- 落盘报告/台账前无工区则先建 `runs/<slug>/outputs/`。
- 用例必须覆盖方案测试范围与开发改动声明；超出或遗漏都需显式说明。

## 跳步替代门禁（用户明示跳过 test）

当前执行者必须补齐：最小回归集、验证证据、残余风险、放行口径、跳过理由。  
缺项不得给放行结论。

## 输入与前置条件

1. 验收口径（Given/When/Then）
2. 开发交付（`developer.implementation.md` + 改动与兼容说明）
3. 工程约束（分层/线程/一致性）
4. 发布约束（窗口/灰度/关键模块）

材料不足时先标“待补材料”并说明信度影响；阻塞项先 `AskQuestion`。

## 文档与实现一致性（流水线）

进入 test 前检查：

1. `outputs/developer.implementation.md` 存在且 §8 reconcile 已勾选。
2. `plan.solution.md` 无与 `git diff` 矛盾的「待实现 / 阻塞 / 条件放行」表述（除非 §7 明确未完成）。
3. `analyze.requirements.md` 覆盖 develop 阶段新增验收口径（如 implementation §3 所列）。
4. 不一致时：阻塞全量放行，退回 develop 补 reconcile 或更新 implementation。

## 测试执行流程

1. 识别主路径、边界、兼容路径
2. 按优先级组织火力（Critical -> High -> ...）
3. 覆盖正向/异常/边界与必要并发弱网场景
4. 自动化可做则做；手测必须留复现步骤与证据
5. 缺陷记录影响范围与复现稳定性
6. 给出放行结论（过/附条件/不过）与残余风险

## 测试准则

- 主干优先于边角
- 结论必须附步骤与证据
- 用户可见变化必须有测试结论
- 兼容路径至少一条实测
- 未测范围必须显式披露

## 报告与台账规则（强制）

1. 每次本 Skill 执行结束必须落盘文档并返回路径
2. 有问题更新 **test** 台账 `outputs/test.ledger.md`
3. 无问题输出 **test** 报告 `outputs/test.report.md`
4. 返回摘要应含：路径、时间、来源、新增问题数、结论
5. `report` 定稿遵循 clean baseline；`ledger` 保留问题追踪历史

## 手动补录问题（人工验收后）

- 先用 `AskQuestion` 采集结构化字段，再写入台账
- 仅允许追加，不覆盖历史
- 新增问题至少包含：ID、标题、严重级别、复现步骤、实际/期望结果、影响范围、状态、证据、记录时间、来源

## 新建与追加判定

- 新建：新版本/新轮次/旧台账已结项/范围不一致
- 追加：同版本同范围且旧台账未结项
- 冲突时默认新建并关联旧台账

## 升级路径

- 实现阻塞：`develop`
- 方案/闸门问题：`review` / `plan`

## 禁止项

- 无证据即放行
- 忽略兼容验证（用户明确要求回滚验证时也不可跳过）
- 仅罗列现象不说明影响与优先级
- 执行结束不落盘
- 手动补录覆盖历史内容
