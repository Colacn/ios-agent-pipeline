# Ruling Criteria

评审执行流程与结论模板见：`execution-playbook.md`。

先决条件（所有等级通用）：

- 阻塞项已经 `AskQuestion` 澄清，或用户明确标注“暂不回答”。
- 结论必须有证据锚点（来源于 `inputs/` 或上游 `outputs/`）。

## 严重度口径（先统一再评级）

- `Critical`：阻塞实现或造成高概率线上事故（安全、数据一致性、崩溃、核心链路不可用）
- `High`：不立刻阻塞但会显著影响正确性/兼容性/可验证性
- `Medium`：可在当前迭代内修复的质量缺口（可观测性、边界覆盖不足等）
- `Low`：优化项，不影响本轮推进

## 量化判定辅助（用于降低主观差异）

- 先计算：`score = 10 * Critical + 3 * High + 1 * Medium`
- 再结合“是否存在未澄清阻塞项”与“证据完整性”决定最终等级

## 通过

关键风险可控，缺口不阻塞实现，验收可执行。通常同时满足：

- 无 Critical 问题；
- High 问题为 0，或已闭环并有证据；
- 不存在“等待用户答复”的阻塞项。
- `score <= 2`，且证据完整（关键结论均有来源锚点）。

## 有条件通过

存在可修复缺口，但明确列出必改条件后可进入实现。通常同时满足：

- 存在 High/Medium 问题但有清晰必改项；
- 必改项可在进入下一步前完成；
- 若有“等待用户答复”项，必须不阻塞当前推进范围。
- `3 <= score <= 8`，且无未澄清的阻塞项。

## 不通过

存在阻塞级风险/证据缺失/方案不可落地，需回到 **plan** 或 **analyze** 补齐。典型触发：

- 有未澄清的阻塞项且用户未给决策；
- 关键结论缺证据或与上游/inputs 冲突；
- 必改项无法在当前轮次收敛。
- 任一 Critical 未闭环；或 `score >= 9` 且短期无法收敛。
