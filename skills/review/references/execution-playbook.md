# Review 执行细则

> 本文件为 **review** Skill 的完整执行规范。与主会话或子代理调用方式无关，执行时以本文为准。

## 定位

不改写需求、不重写方案；在现有材料上给出带证据的评审结论与最小修复方向。

## 流水线中的本步（闸门评审）

在流水线中，本 Skill 对“可落地性、风险、可测性”做闸门判断，并对测试策略可执行性表态。

- 落盘前无工区则先建 `runs/<slug>/outputs/`。
- 证据基线：评审前通读 **analyze** 需求清单 `analyze.requirements.md` 与 **plan** 技术方案 `plan.solution.md`，必要时回读 `inputs/`。
- 每条 Critical/High/Medium/Low 意见都需标注证据来源章节。
- 发现冲突要作为必改项写入结论，不代 `plan` 补方案。

## 评审输入范围

1. 来自 `analyze` 的需求侧材料（含范围与验收口径）
2. 来自 `plan` 的方案侧材料（含第 11 节测试衔接）
3. **develop 后复评**：`developer.implementation.md` 与 `git diff`；plan/requirements 是否已 reconcile（无过时「待实现」表述）
4. 人类补充约束（窗口/合规/性能/回滚要求等）

材料不齐需在结论中标注“待补材料”与信度影响。

## 评审维度与准则

- 风险（缓释与残余）
- 可行性（分层、依赖、验证闭环）
- 安全
- 性能（主线程与热路径）
- 兼容口径
- 可测与可观测
- 测试衔接（第 11 节是否支撑开发执行）

若用户明确要求回滚/处置预案，再补审该维度。

## 评审输出模板

A. 评审结论（通过 / 有条件通过 / 不通过）  
B. 关键发现（分级 + 证据 + 影响 + 最小建议 + 是否阻塞）  
C. 分维度评分（1-5）  
D. 必改项 / 建议项  
E. 待确认（仅阻塞项，完整列出）

## 评审执行顺序

先对齐 `analyze` 与 `plan` 材料，再按维度评审，最后给闸门结论。

## 定稿去痕与澄清

- 定稿去痕、局部修订、过程项清理统一遵循 `references/workflow/pipeline.md`
- 证据缺失、分级无法判定、口径冲突时必须先 `AskQuestion`

## 多视角评审（可选）

可按风险/安全/性能分视角并线，最终归并为统一结论。

## 禁止项

- 无证据下结论
- 不区分阻塞与优化
- 漏看安全/性能/兼容关键维度
- 写成无法执行的散文式输出
