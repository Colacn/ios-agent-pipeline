# Analyze 执行细则

> 本文件为 **analyze** Skill 的完整执行规范。与主会话或子代理调用方式无关，执行时以本文为准。

## 核心原则

1. **只基于用户已提供的信息**：不臆造事实；缺信息标为「未知」或「待用户补充」。
2. **意图与动机**：可做合理推断，但需标注依据与不确定性。
3. **不确定就停步提问**：优先澄清，避免在模糊前提下输出长篇结论。
4. **分析不等于建言**：允许拆解与归纳，禁止方案推荐与优先级裁决。
5. **角度约束**：用户指定角度优先；未指定时做必要的多角度分析。
6. **需求与技术边界（硬性）**：**analyze** 需求清单（`outputs/analyze.requirements.md`）只写需求与验收，不写技术实现细节。

### 《需求清单》定稿体例（推荐默认）

- **目标**：**analyze** 需求清单（`outputs/analyze.requirements.md`）可一眼读完；读者知道「要做什么、几条主线」，不在这里读技术方案。
- **标题**：`# 需求清单` 或团队统一主标题即可；**避免**在标题中堆「移动端 / iOS」等（除非多端正面对比场景）。
- **正文**：`> 来源：…` 一行可定位 PRD/依据即可；**不**强依赖不存在的可检索副产物，**不**写「细节见 `plan`」类元注脚，除非用户明确要求。
- **条目**：用 **1. 2. 3.** 编号列出核心需求，每条业务语义完整；不展开大表。细约束、分档、工程边界由 **plan** 承接并写进 `outputs/plan.solution.md`（**plan** 技术方案）。
- **对下游**：`plan` 必须能与编号条目 **逐条对表**；本清单不在此写类名/接口/状态机。

## 工作流程

1. 承接上下文（区分用户原话与你的理解）。
2. 给出意图/动机初判（带依据与不确定）。
3. 信息不足时优先 `AskQuestion`，可辅以 A/B/C 备选理解。
4. 用户确认后再输出分析；仍缺信息则继续澄清。
5. 待确认项按阻塞程度完整覆盖。
6. 本轮收尾必须交付《需求清单》文档（正文或明确落盘路径）。

## 单题对齐协议（流水线内默认）

1. 按阻塞项提问，先关键后次要。
2. 优先选项式提问，减少开放题负担。
3. 确认即回填并清理已决项。
4. 同主题改口径以最新为准，显式覆盖旧表述。
5. 阻塞性口径未对齐，不进入 plan 阶段。
6. 定稿去痕 + 局部修订：最终文档只保留生效结论，不保留决策流水账。
7. 先问后写：阻塞项未确认不得用“未决清单”替代结论（用户明确延后除外）。

## AskQuestion 主动澄清规则（硬性）

1. 触发条件：目标不清、范围不清、验收缺失、信息冲突、术语多义等。
2. 优先选项化并覆盖全部关键阻塞项。
3. 最小覆盖：目标口径、范围边界、验收标准。
4. 未确认前不输出确定性结论，不进入下游。
5. 禁止无关追问与一次性大量开放题。

## 输出结构（按轮次选用）

### 轮次 A — 待澄清

- 当前理解
- 意图与动机假设（依据 + 不确定）
- 待你确认（问题清单或多种可能）
- 本轮不展开深度分析原因

### 轮次 B — 分析交付

- 分析主题与边界
- 用户指定角度或多角度分析
- 信息矛盾/缺口（只指出，不给建议）
- 待确认（仅保留未决项）

## 输入源与解析范围

1. 纯文本 / 对话摘录
2. `md`（按结构提取可引用条目）
3. `docx`（读正文与表格；失败则说明并索要可复制文本）
4. URL（抓要点；不可访问则说明并请求关键片段）

## 禁止项

- 行动建议、路线推荐、价值判断式断语（除非用户明确要求并先完成工作模式切换确认）
- 无来源的外界事实断言
- 在非需求包场景强套需求八节

## 跨会话与落盘（可选）

多轮场景可同步规划文件；写入仓库前遵循 `AGENTS.md` 的执行前确认。

## 流水线中的本步（详细版）

在走流水线时，本 Skill 负责把“要做什么、证据在哪、边界是什么”对齐清楚。首轮要做外部依据判断并尽量脚本化归档。

- 证据基线：以 `runs/<slug>/inputs/` 为唯一事实源，读齐后再产出结论。
- 外部依据判定：区分“有本地路径/URL”与“仅自然语言”。
- slug 选择：先看分支；分支语义不一致则先与用户确认。
- runs 门禁：归档未完成不得落盘 `outputs/`。
- 归档诚信：禁止用 TODO 文案替代实际 ingest。

## 附录 A（可选）：规格化需求包八节

仅在用户明确要求“需求包/给 plan 侧交接”时启用：

1. 需求简述
2. 范围定义
3. 功能需求
4. 非功能
5. 用户故事与交互（如适用）
6. 验收标准
7. 待确认问题（仅未决项）
8. Task Draft
