# 假设显式化协议（Analyze）

> 借鉴 spec-driven 工作流：在定稿前把隐性假设写出来，避免「以为用户要的是 X」类返工。  
> 完整 SOP 仍以 `execution-playbook.md` 为准。

## 何时触发

- 需求含糊、范围未闭合、验收不可测
- 用户描述含「更快」「更好」「简单改一下」等未量化词
- 涉及平台、鉴权、存储、合规等未明示的技术前提

## ASSUMPTIONS 块（定稿前必出）

在输出确定性结论或写入 `analyze.requirements.md` 前，先列出假设并请用户确认或纠正：

```markdown
## 我在做的假设（请纠正）

1. 目标平台为 [iOS/Android/Web/…]（依据：…）
2. 鉴权方式为 […]（依据：…）
3. 本轮不改 [范围外项]（依据：…）
→ 若有误请现在指出；未纠正则按此推进。
```

未获确认前：标为「待确认」，不进入 **plan**。

## 成功标准改写

把模糊需求改写成可验收条件，再请用户确认：

```text
原述：「让列表更流畅」
改写：
- 首屏可交互 < 1s（设备/网络口径写明）
- 滚动无明显掉帧（定义 FPS 或主观标准）
→ 这些是正确目标吗？
```

改写结果进入需求清单的验收条款，而非留在对话里。

## 边界三档（写入需求清单或交接段）

| 档位 | 含义 | 示例 |
|------|------|------|
| **Always** | 本轮默认必须做 | 改代码前跑项目约定测试命令 |
| **Ask first** | 须先问用户 | 加依赖、改 schema、动 CI |
| **Never** | 禁止 | 提交密钥、删失败测试换绿 |

## 常见借口（勿跳过）

| 借口 | 事实 |
|------|------|
| 「用户很清楚要什么」 | 隐性假设是返工主因；两行假设清单成本极低 |
| 「先分析再补假设」 | 假设未清时的「分析」会固化错误方向 |
| 「太简单不用写验收」 | 简单任务也要 1–3 条可测验收 |

## 红旗

- 无假设块却输出长篇定稿
- 验收只有形容词、无量化或 Given/When/Then
- 把技术方案写进 `analyze.requirements.md`

## 出口核对

- [ ] 假设已列出且用户已确认或标注「暂不回答」
- [ ] 阻塞项未清不进 **plan**
- [ ] 需求清单可逐条对表验收
