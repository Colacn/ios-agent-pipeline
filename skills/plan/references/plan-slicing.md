# 任务切片与规划纪律（Plan）

> 借鉴 planning-and-task-breakdown：纵向切片、可验证小任务、阶段检查点。  
> 与 `plan-checklist.md`、`risk-gate.md` 配合使用。

## 规划模式（先读后写）

进入 **plan** 时默认只读：读 analyze 产出、相关代码、overlay 分层；**本阶段产出是方案文档，不是实现**。

## 依赖与顺序

先画「谁依赖谁」，再定实现顺序——自下而上或**纵向切片**二选一，避免纯水平分层：

```text
差：先做完所有 DB → 再做所有 API → 再做所有 UI
好：切片 1 = 用户能完成核心动作 A（DB+API+UI 最小闭环）
```

## 任务粒度

| 规模 | 约文件数 | 说明 |
|------|----------|------|
| XS–S | 1–2 | 单点改动，单会话可完 |
| M | 3–5 | 一条纵向切片 |
| L | 5–8 | 考虑再拆 |
| XL | 8+ | **必须拆分** |

单任务验收点建议 ≤3 条；标题里出现「并且」通常是两个任务。

## 任务模板（写入 plan.solution.md）

```markdown
### Task N：[短标题]

**验收：**
- [ ] …
- [ ] …

**验证：** 测试/构建/人工检查命令（写全，不只写「跑测试」）

**依赖：** Task … / 无

**可能改动：** `path/…`（预估，非承诺）
```

## 阶段检查点

每 2–3 个任务插入检查点：

```markdown
### Checkpoint：Phase 1 后
- [ ] 测试通过、构建干净
- [ ] 核心路径端到端可用
- [ ] 人类已看过再继续（L2+ 或高风险时）
```

## 常见借口

| 借口 | 事实 |
|------|------|
| 「任务显而易见不用写」 | 写下来才能暴露隐藏依赖与边界 |
| 「规划是浪费时间」 | 无计划实现只是在打字；10 分钟规划省数小时返工 |
| 「可以边做边拆」 | 上下文会丢；书面计划可跨会话 |

## 红旗

- 无任务列表就开始 **develop**
- 任务无验收或无验证步骤
- 全是 XL 任务、无检查点

## 出口核对

- [ ] 每条需求在 §0 / 正文有落点
- [ ] 任务有验收 + 验证 + 依赖顺序
- [ ] 高风险项靠前（fail fast）
