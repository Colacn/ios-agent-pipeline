# Plan 执行细则

> 本文件为 **plan** Skill 的完整执行规范。与主会话或子代理调用方式无关，执行时以本文为准。

## 输入与前置条件

- 首选输入：`analyze` 侧对齐材料或规格化需求包。
- 边界承接：**analyze** 需求清单 `analyze.requirements.md` 是需求与验收输入，技术细节在 **plan** 技术方案 `plan.solution.md` 完成。
- **极简清单时**：若 `analyze` 只余若干条**编号需求**，本方案须含 **§0 对表** + 各条在**后文**设计/范围/测试等节中的**可验收落点**（见 `plan` Skill 第 8 条；**不要**在文首用整节 **§0.1** 与后文**重复**铺陈），把约束写全，避免「需求瘦身 → 方案缺口」。
- 关键信息缺失（兼容窗口、依赖约束、落层争议）时，标「待确认」并按阻塞程度覆盖。

## 方案规划原则

1. 贴着现有架构小步演进，确保可验证。
2. 先分层再类名，保持单向依赖，禁止绕过既有主链路。
3. 命名与仓库习惯一致，不随意发明新分层术语。
4. 先粗读后细追：规则 + 目录 + 真实调用链。
5. 与上游分析对表，决策与落点可回映。
6. 附录 A 可小步维护，但改条文须先征得用户同意。

## 技术方案正文结构

1. 方案概述
2. 现状与上下文
3. 目标架构与分层推演（必填）
4. 关键设计决策
5. 改动范围
6. 兼容、迁移与发布处置（按需；回滚预案仅用户明确要求时补）
7. 非功能
8. 风险登记册
9. 任务拆分建议
10. 待决策 / 待确认（仅未决阻塞项）
11. 测试策略与验证衔接（流水线必填）

**与《需求清单》的衔接**（`analyze` 为**编号短清单**时建议置于正文前部，名从「§0 与《需求清单》对应关系」等即可）：

- **索引表**（需求第 N 条 → 本方案章节 / 任务包 / 阻塞项）**必填**；**可验收细则**写在**对应后文章节**（决策、范围、测试等），**不要**在 §0 下再开一整节与正文**重复**的「细则承接区」（见 `SKILL.md` 第 8 条「正文为真源，不单拧 §0.1」）。
- 凡从 PRD 收敛后仅体现在方案内部、未写入短清单的要点，须在**其所属设计/范围/测试小节**中写明，供 `review` 稽核。
- **不要**在「对表/索引」前单写「从 PRD 迁入 / 需求清单不逐条展开」类元话；见 `SKILL.md` 第 8 条「元说明禁写」。

## 分析与交付流程

1. 先看附录 A、目录与调用链。
2. 落层未定前不写类级细节。
3. 存在阻塞项时先 `AskQuestion`。
4. 默认不写实现代码（除非用户明确要求）。
5. 本轮收尾必须交付《技术方案》文档（正文或落盘路径）。
6. 通用门禁（定稿去痕/局部修订/先问后写）遵循 `references/workflow/pipeline.md`。

## AskQuestion 主动澄清规则（硬性）

触发条件与流程遵循 `references/workflow/pipeline.md`；本 Skill 重点澄清分层落点、依赖约束、验收可评审性。

## 流水线中的本步（详细版）

- 证据基线：写方案前通读 `inputs/` 与 **analyze** 需求清单 `outputs/analyze.requirements.md`，冲突项先回问。
- 前置：关键技术决策已有结论或被标注待确认。
- 必须项：文件清单、实现路径、验收标准明确；第 11 节完整可执行。
- 推进：完成后请求进入 `review` 阶段。

## 分层条文维护（附录 A）

按“问题归类 -> 变更提案 -> 影响评估 -> 验收方式”推进；改共享底座前必须先征得用户同意。

## 输出自检清单

- 与上游范围/验收是否对齐
- 分层与依赖是否一致
- 兼容口径是否明确（用户明确要求时再补回滚预案）
- 风险与非功能是否可验证
- 第 11 节是否完整
- 是否已交付《技术方案》文档

## 禁止项

- 未读分层与调用链即做大重构
- 以“业界最佳”覆盖本项目约束
- 与上游材料无法对表
- 未经用户同意改附录 A 或共享底座
