# 增量交付纪律（Develop）

> 借鉴 incremental-implementation：薄切片、每步可测、范围克制。  
> 与 `collaboration-discipline.md`、`delivery-checklist.md` 配合。

## 增量循环

```text
实现最小切片 → 跑测试/构建 → 验证 → 记录 → 下一切片
```

每切片结束后项目须**可构建、既有测试仍绿**（除非任务明确要求红字 TDD 阶段）。

## 切片策略（优先纵向）

- **纵向**：一条用户路径走通（存储+逻辑+入口最小集）
- **契约先行**：API/类型定稿后前后端可并行
- **风险先行**：最高不确定项先做，失败早暴露

## 范围纪律

只改任务要求范围内的文件。发现旁路问题**不要顺手改**：

```markdown
## 注意到但未动（可选后续任务）

- `path/foo` 有无用 import（与本任务无关）
- …
→ 需要单独开任务吗？
```

## 简洁性自检（提交前）

- 能否用更少行达到同样行为？
- 抽象是否超过第三次复用需求？
- 是否在为一两个用例造通用框架？

## 单切片核对

- [ ] 本切片只做一件事
- [ ] 已跑项目约定测试/构建（有改动才重跑；**未改代码不重复刷同一命令**）
- [ ] `developer.implementation.md` 已更新改动与验证
- [ ] 建议 `bash scripts/reconcile-check.sh <slug>`

## 常见借口

| 借口 | 事实 |
|------|------|
| 「最后一起测」 | 切片 1 的错会让后面全错 |
| 「顺手重构一下」 | 混在功能里难 review、难回滚 |
| 「再多跑一遍测试更放心」 | 代码未变时重复跑无新信息 |

## 红旗

- 单次会话 >100 行且未跑验证
- 多无关改动混在一次提交
- 评审阻塞未解除仍改代码
- 跳过 reconcile / 改动清单

## 出口核对

- [ ] 改动清单 + 验证结果 + 残余风险
- [ ] 出口文档与仓库实际 diff 一致（reconcile）
